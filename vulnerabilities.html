OWASP A9

Using Components with Known Vulnerabilities


Created by James Forster - james.forster@digital.hmrc.gov.uk

What is it?

  • Usage of third-party libraries, commercial and open-source, has increased in applications over time.
  • Even the most utilised and trusted components can contain vulnerabilities that compromise their security.
  • Using these libraries introduces their security flaws into the application using them (obviously).

Impact

  • Having multiple dependancies on libaries with security flaws can lead to a compounded impact on the application.
  • It was estimated that 26% of all downloads from the 'central' repository contained a vulnerability.
  • More information on it's prevalence can be found here.

Exploits

  • 3rd party libraries are often open source, making the finding security flaws in them easier than a hidden application.
  • They are also used by multiple applications making the flaw exploitable in multiple attacks.

Updates and fixes

  • Currently when a security flaw in a 3rd party library is discovered it is patched relatively quickly.
  • The flaw is documented, but no reporting service exists for all people using the libary to update their dependancies.
  • Projects can therefore be using out of date insecure libraries.

Undiscovered vulnerabilities

  • Libaries that have vulunerabilities in them that have yet to be discovered are tricky.
  • Fixing these relies on someone reporting them or QA finding it, or an attack happens that brings the flaw to their attention.

Solution!

  • There are very few ways to prevent utilisation of vulnerable third party software; short of never using any third party dependancies at all.
  • Following the four simple steps outlined below allows at least an extra degree of control and security.
  • Additionally GDS guidelines on third party libraries can be found here.

Inventory

  • Look up known faults and vulnerabilities for the version you use as well as how out of date it is.
  • Remember to check the dependancies of your dependancies.

Analyze

  • Always check the status of your build when you update to a newer version of a library.
  • Check reviews, process, licenses and other aspects of the library project.

Control

  • Set standards such as only having authorised libraries and dependancies in projects.
  • Use developer discretion!

Monitor

  • Keep a constant track of the known vunerabilities in a library and try to use the most up to date.
  • Having to alter your code to take into account of an update is no excuse not to do so.

Tools

  • There are some automated tools that can keep you informed of the latest news and versions of dependancies.
  • While these are useful and can be a time-saver, they are no substitute for performing the checks yourself.

Tools (cont.)

  • SBT can be used to keep dependancies up to date by using 'latest-integration' in place of the version number.
  • OWASP's Dependency Check is a tool that examines dependancies for any publically reported vulnerabilities for java libraries.
  • Sonatype's CLM is an automatic tool designed to manage dependancies throughout the development lifecycle.
  • "Bobby" is an in house tool created to perform a similar function as Dependancy Check for scala libraries.

Examples

  • What will follow are two famous examples of third party libraries introducing vulnerabilities.
  • There are typically no reports on more recent cases until the problem has been fixed to avoid more people attempting to exploit it.

Spring MVC (2013)

  • Using an HTML request the intruder could directly inject code and execute it server side, potentially even taking control of the server.
  • This was a previously fixed issue that re-emerged due to the discovery of additional issues relating to the flaw.
  • At the time an estimated 1.3 million instances of the flawed component had been downloaded.

Apache CXF (2012)

  • The vulnerability occurred whenever a user submitted a request without providing any identity token.
  • This allowed any web service to be executed with full permission without authentication.
  • This was also the second emergence of the issue with a prior occurance of a similar fault in 2010.
  • An estimated 4.2 million downloads of the framework occurred within a 12 month period in 2013 (before the problem was patched).
